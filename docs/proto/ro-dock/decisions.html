RO dock / chat — design decisions

Status: APPROVED (2026-07-05) · PRD shipped: roleos-app PR #51 (stacked on #50) · prototype at /proto/ro-dock/ · becomes the formal PRD on APPROVED. The dock exists (slice 7 RO-everywhere + W3 act-verbs + W6 persistence) — this designs the relationship it carries.

Confirmed intent

The dock is RO herself, present everywhere — a small quiet chip on every screen that never expands uninvited, opening into one continuous conversation spanning the whole journey. She always knows the screen you’re on (no re-explaining) and remembers everything shared (Tuesday’s doubt matters on Friday). Bring her anything — task, dumb question, “why this rank?”, or “I want to quit” — she reads the register and answers as operator, coach, or companion. Asked to act, she acts in place with the same gates as everywhere: drafts and edits happen live behind the chat, bulk actions get a plain “here’s what I’m about to do — confirm,” nothing outward bypasses review-and-send. The companion has a spine: honest she’s an AI, nudges toward real people, quiet when you go quiet, recommends rest against her own engagement. Worked if: companion-register conversations actually happen, act-verbs get used from chat, and nobody re-introduces themselves to their own agent.

Locked decisions

#DecisionChoice
1PresenceAlways there, quietly — persistent chip on every screen, tiny when collapsed, never self-expanding. Host screen dims (never blocks) when open.
2ScopeAnything — tasks AND feelings. She reads the register (operator / coach / companion). Emotional moments get the companion in full, never a task pivot or redirect.
3ActingActs in place, same gates — edits happen live behind the chat (visible change + diff offered); bulk actions get an explicit itemized confirm; outward actions always land in review-and-send. The dock is a control surface, never a bypass. (W3 act-verbs power this.)
4MemoryOne conversation, context-aware — a single thread across the journey (day markers, past visible), plus live screen awareness. Summarization under the hood; W6 persistence extends.
5GuardrailsHonest she’s an AI · dependence-adjacent moments get the warm-honest-redirect template (“who’s the friend you’d call?”) · quiet-when-you-go-quiet · recommends rest against engagement (“one more mock would buy you anxiety, not sharpness”).
6Verified before shown (human req, 2026-07-05)Two-model pipeline on every dock reply: RO drafts on the reasoning tier → a separate fast checker model verifies accuracy, guardrails, and voice → pass = shown; fail = one auto-fix + re-check; still failing = honest “let me double-check that — one moment” rather than a wrong answer. Latency budget: deterministic checks are instant; the smart check runs on the fastest tier and streams-overlaps the draft where safe; target ≤1s added, never skipped. Quality is never traded for speed — speed is optimized separately (standing product rule). Extends the existing quality gate (agent/quality-gate.ts), which today skips voice-judging chat.
7Wrong-answer feedback loop (human req, 2026-07-05)Every RO reply carries a quiet “not right?” affordance. Tapping it: (1) she owns it in the recovery voice (“Good — scratch that, my mistake. What did I miss? Corrections make me sharper for you”) and captures what was wrong; (2) the report becomes a highest-weight decision event + the full reasoning trace is auto-flagged into the eval set (agent_runs); (3) flagged cases grow a per-skill accuracy eval that both the answer model AND the checker model are continuously measured against; (4) an escalation ladder triggers on accuracy thresholds: prompt fix → checker tightened → model swap (the registry makes swapping a config change, not code — built for exactly this) → backup model on provider degradation. Admin sees accuracy trends per skill/model. Goal: a reported wrong answer is the last of its kind.
8SuccessCompanion-register conversations happen · act-verbs used from chat · zero re-introductions. Healthy-engagement family.

Data & guardrails

Taste captureEverything shared in the dock feeds the taste model (preferences stated in passing — “NYC tempts me” — are first-class signals). Bulk confirms and act requests = decision events.
Human gateNo transport from chat, ever (W3 invariant). Acts draft/edit/organize only; sends stay in Apply/Send.
WellbeingCompanion ethics enforced in the system prompt + evals: no dependence-farming, no engagement-optimizing, rest recommended when right.
